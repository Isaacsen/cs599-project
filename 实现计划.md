根据当前新架构，项目应按 **“软件工程师 Agent + 权限隔离执行平台”** 来推进，而不是单纯的自动测试工具。推荐实现步骤如下：

**第一阶段：SDD 规格与项目定位**

1. 明确项目定位：面向 Python 项目的软件工程师 Agent。
2. 更新三份规格文档：
   - `Product Spec`：代码审查、自动修 Bug、生成单测、LLM 生成、权限隔离。
   - `Architecture Spec`：Agent 编排、工具层、沙箱层、可观测层。
   - `API Spec`：所有 CLI 输入输出与 JSON 数据结构。
3. 更新 `README.md` 和 `docs/architecture.md`，保证课程要求中的 SDD 和架构图完整。

**第二阶段：基础工具层**

1. 实现 `Repo Scanner`：扫描 Python 文件、测试文件、忽略缓存和虚拟环境。
2. 实现本地测试执行器：调用 `python -m pytest`。
3. 实现 JSON Report Writer。
4. 建立最小闭环：

```
python -m src.main examples/sample_python_project
```

目标：能扫描项目、执行已有测试、输出结构化结果。

**第三阶段：权限隔离执行层**

1. 实现 Docker 沙箱执行器。
2. 加入安全策略：
   - 只读源码挂载。
   - 禁用网络。
   - CPU / 内存 / PID 限制。
   - 超时控制。
   - `no-new-privileges`。
3. 支持：

```
python -m src.main examples/sample_python_project --executor docker
```

目标：所有生成代码都可以在受限环境中执行。

**第四阶段：测试规划与规则型测试生成**

1. 实现 

   ```
   Test Planner Agent
   ```

   ：

   - AST 分析公开函数。
   - 生成 `TestPlan`。

2. 实现规则型 

   ```
   Test Generator Agent
   ```

   ：

   - 根据函数名和场景生成 pytest。

3. 实现临时测试工作区，避免修改原始项目。

4. 实现 `Security Checker Agent`，拦截危险 import / call。

目标命令：

```
python -m src.main examples/sample_python_project --generate-tests --executor docker
```

**第五阶段：结果分析、失败诊断与评估**

1. 实现 `Result Analyzer Agent`：解析 pytest 输出。
2. 实现 `Failure Diagnoser Agent`：给出失败类型和修复建议。
3. 实现 Benchmark：

```
python -m src.benchmark --executor docker --output docs/runs/benchmark.json
```

目标：满足课程中的可观测性与评估要求。

**第六阶段：代码审查 Agent**

1. 实现 `Code Reviewer Agent`。
2. 检测：
   - `eval` / `exec` / `subprocess` 等危险调用。
   - 疑似硬编码密钥。
   - 宽泛异常。
   - 缺失测试覆盖。
   - 除零风险。
3. 输出：

```
python -m src.review examples/review_target --output docs/runs/review.json
```

**第七阶段：自动修 Bug Agent**

1. 实现 `Bug Fixer Agent`。
2. 默认 dry-run，不直接改源码。
3. 支持安全修复：
   - `eval` -> `ast.literal_eval`
   - 硬编码密钥 -> `os.getenv`
   - `except Exception` -> 更窄异常
   - 除零保护
4. 命令：

```
python -m src.fix examples/review_target --output docs/runs/fix_plan.json
```

需要写回时才使用：

```
python -m src.fix examples/review_target --apply
```

**第八阶段：缺失覆盖单测生成 Agent**

1. 实现 `Unit Test Writer Agent`。
2. 分析已有测试，跳过已覆盖函数。
3. 生成 pytest 内容。
4. 默认 dry-run，显式 `--apply` 才写入测试文件。

```
python -m src.unit_tests examples/review_target --output docs/runs/unit_tests.json
```

**第九阶段：软件工程师 Agent 总控编排**

1. 实现 `Software Engineer Agent`。
2. 串联：
   - Code Review
   - Bug Fix Plan
   - Unit Test Writer
3. 输出统一报告：

```
python -m src.engineer examples/review_target --use-llm-review --use-llm-tests --run-sandbox --sandbox-executor docker --output docs/runs/software_engineer.json
```

目标：项目核心 Demo 应以这个命令体现“软件工程师 Agent”。

**第十阶段：LLM 测试生成 Agent**

1. 实现 LLM Config：

   - 默认 DashScope。
   - 支持 DeepSeek。
   - API Key 只从 `os.getenv()` 读取。

2. 实现 OpenAI-compatible LLM Client。

3. 实现 

   ```
   LLM Test Generator Agent
   ```

   ：

   - 构造 Prompt。
   - 调用真实 LLM provider。
   - 提取 Python 代码块。
   - 经过 Security Checker。

4. 命令：

```
python -m src.llm_tests examples/sample_python_project --output docs/runs/llm_tests.json
```

LLM 测试生成默认使用真实 Agent 调用。

**第十一阶段：Demo 与课程交付物**

1. 更新 `scripts/run_demo.ps1`，串联所有核心命令。

2. 确保 

   ```
   docs/runs/
   ```

    包含：

   - `sample_run.json`
   - `benchmark.json`
   - `review.json`
   - `fix_plan.json`
   - `unit_tests.json`
   - `software_engineer.json`
   - `llm_prompt.json`
   - `llm_tests.json`

3. 更新 `docs/demo_guide.md`。

4. 更新 `docs/artifacts.md`。

**第十二阶段：最终报告与提交**

1. 完成 `docs/CS599_大作业报告.md`。
2. 转为带目录导航的 PDF。
3. 检查：
   - 无明文 API Key。
   - 测试通过。
   - Demo 可运行。
   - README 完整。
4. 最终提交并推送。
